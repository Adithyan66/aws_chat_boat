# Database Design (MongoDB)

Since the chatbot interaction is conversation-based and variables extracted via AI might evolve dynamically or consist of unstructured payloads, a NoSQL database like MongoDB is highly suitable.

## Collections

### 1. `chats` (Conversation & State Management)
Tracks ongoing conversations, message history, and the current state of a resource creation pipeline.

**Schema**:
- `sessionId` (String, Indexed): Identifier for the user/session.
- `history` (Array of Objects): 
  - `role` (String): `user` or `assistant`.
  - `content` (String): The message text.
  - `timestamp` (Date)
- `currentState` (Object):
  - `intent` (String): e.g., `CREATE_EC2`, `CREATE_S3`, or `NONE`.
  - `collectedData` (Object): Key-value pairs (e.g., `{ region: 'us-east-1', instance_type: 't2.micro' }`).
  - `status` (String): `PENDING_VARIABLES`, `READY_FOR_TERRAFORM`, `COMPLETED`.

**Reasoning**: Storing `currentState` in the chat document avoids losing context between API calls. When a user replies with "us-east-1", the backend matches this to the `PENDING_VARIABLES` state and updates `collectedData`.

### 2. `resources` (Created AWS Resources)
Stores successfully planned or validated resources generated by the chatbot.

**Schema**:
- `sessionId` (String, Indexed): Owner of the resource.
- `resourceType` (String): `EC2` or `S3`.
- `name` (String): Name of the instance/bucket.
- `region` (String): AWS Region.
- `details` (Object): Specific details (e.g., `instanceType`).
- `terraformCode` (String): The actual `.tf` text generated.
- `status` (String): `VALIDATED`, `PLANNED`, `FAILED`,`COMPLETED`.
- `createdAt` (Date).

**Reasoning**: Searching natural language queries like "Show me all EC2 instances" involves mapping the query to a database search on the `resources` collection filtering by `resourceType = 'EC2'`.
